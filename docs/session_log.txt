SESSION 007 — 2026-05-19
--------------------------------------------------------------------------------
WHAT
  Fixed two audio/latency bugs: TTS stuttering on Windows and a ~1 second delay
  between wake word detection and HADES's first spoken response.

  BUG FIX — PIPER TTS STUTTERING ON WINDOWS
  - Root cause: Piper's alan-medium voice outputs at 22050 Hz (int16). Windows
    audio devices default to 44100 or 48000 Hz. Passing mismatched rates to
    sd.play() forces the driver to resample internally, producing audible stutters
    and dropouts.
  - Fix: _speak_piper_api() in voice.py now queries the output device's native
    sample rate via sd.query_devices(kind='output') and, when it differs from
    Piper's rate, resamples the audio using np.interp before playback.
  - Also converts int16 → float32 (sounddevice's native type) and guards against
    odd-byte chunks from the synthesizer that would corrupt int16 alignment.

  BUG FIX — ~1 SECOND WAKE RESPONSE DELAY
  - Root cause: hades_loop() in main.py called time.sleep(0.3) immediately after
    wake word detection, before invoking speak("Yes, Sir?"). The 300 ms dead time
    added on top of Piper synthesis time made the system feel unresponsive.
  - Fix: both sleep(0.3) calls removed (sleeping-wake and standby-wake branches).
    Piper synthesis now begins immediately after wake word detection.

  ADDITIONAL AUDIO IMPROVEMENT — REMOVE PER-CALL MIC CALIBRATION
  - listen() previously called adjust_for_ambient_noise(duration=0.5) on every
    invocation, adding a 500 ms silent gap at the start of each listening window.
  - Removed: dynamic_energy_threshold=True already adjusts the energy threshold
    continuously during recognition, making per-call calibration redundant.
  - wait_for_wake_word() and listen_for_wake_word_once() calibration reduced
    from 0.3 s / 0.2 s to 0.1 s each.

SESSION 006 — 2026-05-19
--------------------------------------------------------------------------------
WHAT
  Fixed two bugs: sleep via text input not updating the UI, and screen vision
  responses sounding like a generic third-party AI rather than HADES itself.

  BUG FIX — SLEEP NOT UPDATING UI VIA TEXT INPUT
  - handle_text_command() in main.py had no sleep word detection; it always
    routed to route() and reset status to "standby" regardless of input.
  - Added _text_state dict (closure-scoped) with a "sleeping" bool to track
    sleep state across text commands independently of the voice loop.
  - When a sleep word is typed: _pending_state is cleared, orb enters SLEEPING
    state, goodbye is spoken, the function returns early (no standby reset).
  - Any subsequent text command wakes HADES back to standby before routing.
  - Voice loop sleep state (_sleeping) is unchanged; the two paths are independent.

  BUG FIX — VISION RESPONSES SOUNDING LIKE A THIRD-PARTY AI
  - vision.py sent no system prompt to the Groq Llama 4 Scout API; the model
    responded as a generic assistant and sometimes denied it could see anything.
  - Added a HADES persona system prompt to the vision API call: tells the model
    a screenshot has been captured and attached, to describe it directly, never
    claim it cannot see the screen, respond as HADES, address the user as Sir,
    keep responses to 2-3 sentences, no markdown or bullet points.
  - Rephrased the user-facing prompt in main.py route() from a passive
    "please help them based on what's on screen" to an imperative
    "Describe what you see in the attached screenshot and help the user with
    their request: '...'" — reduces model refusal when the system prompt alone
    is not enough.

  Implemented true sleep/standby mode. Previously, saying "sleep" or "goodbye"
  broke out of the inner command loop but the app continued processing arbitrary
  speech via wait_for_wake_word() without any GUI feedback. Fixed:
  - Sleep now shows a distinct SLEEPING orb state (near-dark, 15% brightness,

  - SLEEP_WORDS expanded to include STT spacing variants ("good bye", "good night",
    "standby", "goodnight", "go to sleep")
  - Added listen_for_wake_word_once() helper to voice.py for future use
